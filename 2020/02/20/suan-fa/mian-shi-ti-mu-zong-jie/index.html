  1. static关键字的用法有哪些?

    • 修饰局部变量，修改了变量的生命周期，在函数第一次被调用的时候被初始化，之后会一直保存这个值，有点类似全局变量了。

    • 修饰全局变量，修改了变量的存储区域，new出来的变量会存在堆中，局部变量存在栈中，而静态变量和全局变量存在全局存储区域中。静态全局变量在main函数之前就会分配空间，如果有初始值使用初始值初始化，否则使用系统默认值初始化。

    • 修饰全局函数，该函数只能在该文件中被访问，可以用来区分多个命名空间中的同名函数。

    • 修饰成员变量，所有对象共享该变量。且不需要生成对象就可以访问该变量。

    • 修饰成员函数，不用生成对象就可以访问该函数，但是在静态成员函数内，只能访问静态变量。

      static关键字的作用?

    • 对作用域进行隐藏，编译多个文件的时候，没有加static的变量全局可见，加了static的对象只能当前文件可见。

    • 保持变量的持久化，存储在静态区域的变量在程序该开始运行的时候就会完成初始化，也是唯一初始化。

    • 默认初始化为0.

    • 所有静态成员函数和静态成员变量都是属于类的，所有对象只有一个拷贝，所以不能将静态成员函数设置为虚函数，虚函数是用来是实现多态的，静态成员函数没有对象的概念，也就无法实现多态。

  2. 继承和组合的的概念，什么时候用继承，什么时候用组合?

    • 继承是通过扩展已有类来获得新的类的方法。
    • 组合是有已有类的对象构建新的类的方法。
    • 继承强调’是’的关系，并且一个类要对另一个类公开所有的接口。
    • 组合强调’有’的关系。
    • 注意，如果没有特殊需求使用组合的方式，如果使用继承就需要将类的所有东西都继承。

  3. c++菱形布局会发生什么，画出对应的内存布局?

    • c++菱形继承是指，有两个类b，c派生自同一个类a,同时d类继承b，c两个类。
    • 出现问题，如果基类中有一个方法，则在d类中保存了两个该方法，如果调用就会出现指向不明确的问题。
    • 解决方法，一个是使用域限定解决，指定所要调用的是哪个类中的函数。另一个解决方法就是使用虚继承。
    • 内存布局。虚继承中多了一个vbptr，vbptr指的是虚基类表指针（virtual base table pointer），该指针指向了一个虚表（virtual table），虚表第一部分存储的是对象相对于存放vptr指针的偏移量；第二项是vbptr到共有基类元素之间的偏移量。

  4. 说说对智能指针的了解?

    • RAII机制(背景知识)：定义一个类来完成资源的释放和分配，在构造函数中完成资源的分配和初始化操作，在析构函数中完成资源的清理工作。从而保证资源的正确初始化和释放。智能指针是一种RAII机制的应用，所以智能指针本质上可以看做是一个类。将基本类型指正封装为类对象指针，并在析构函数中编写delete语句删除指针对象所指向的空间。

    • c++智能指针简单介绍

      • STL提供了四种智能指针auto_ptr,unique_ptr,shared_ptr,weak_ptr。

      • auto_ptr是c++98提供的解决方案，c++11已经弃用。

      • 使用注意点

        • 所有智能指针都有一个explicit构造函数，以指针作为参数，所以不能自动将指针转化为智能指针，必须显示调用。

            template<class T>
  class auto_ptr
  {
      explicit auto_ptr(T *p=0);

      ...
  };

  shared_ptr<double> pd;
  double *p_reg = new double;
  pd = p_reg;                             //not allowed(implicit conversion)        
  pd = shared_ptr<double>(p_reg);         //allowed(explicite conversion)        
  shared_ptr<double> pshared = p_reg;     //not allowed(implicit conversion)        
  shared_ptr<double> pshared(p_reg);      //allowed(explicite conversion)

        • 对所有智能智能都应该避免在创建对象的时候不是在堆上创建对象。当ptr过期的时候程序会将delete作用在非堆的内存，会出现错误。

            string str("hello world");
  shared_ptr<string> ptr(&str);

    • 为什么摒弃auto_ptr?

        auto_ptr<string> ps(new string("hello world"));
  auto_ptr<string> vocation;
  vocation = ps;
      • 如果两个指针是常规指针，则两个指针将指向同一个string对象。所以当程序将删除同一个对象两次。解决方法如下：
        • 定义赋值运算符，将之执行深赋值，这样两个指针将指向不同的对象。缺点是会浪费空间，所以只能指针都没有采用这种策略。
        • 创建所有权的概念，对特定的对象，只有一个智能指针可以拥有。这样只拥有对象的智能指针的构造函数会删除该对象，这是auto_ptr和uniqie_ptr的策略，但是unique_ptr的策略更加严格。
        • 创建智能更高的指针，跟踪引用特定对象的智能指针数，这称之为引用计数。这也是shared_ptr采用的策略。
      • 为什么放弃auto_ptr，避免潜在的内存崩溃的问题，如果将一个auto_ptr赋值给另一个auto_ptr,若在使用原来的auto_ptr就会发生运行期间错误(访问了空的指针)，但是使用unique_ptr不会出现这个问题，因为无法编译通过。

    • unique_str为什么优于auto_ptr?

      • unique_ptr不运行将一个智能指针赋值给另一个智能指针，防止出现上面的编译问题。

      • 当编译器将一个unique_ptr赋值给另一个时，若原来的unique_ptr是一个临时的右值，编译器允许这样做，但是如果unique_ptr存在过一段时间，编译器将禁止这样做。

          unique_ptr<string> pu1(new string("hello world"));
  unique_ptr<string> pu2;
  pu2 = pu1;                  //not allowed
  unique_ptr<string> pu3;
  pu3 = unique_ptr<string>(new string("you")); // allowed

    • 如何选用智能指针?

      • 如果程序使用多个指向同一个对象的指针，应该选择shared_ptr。
      • 如果程序不需要多个指向同一个对象的指针，可以使用unique_str。

  • 虚函数的实现机制?

    • 什么是虚函数：
      虚函数是在类中有virtual关键字声明的成员函数，并且每一个含有虚函数的类都至少有一个与之对应的虚函数表，其中存放着该类所有虚函数对应的函数指针。

        class A{{
      int a;
      virtual void foo();
      virtual void bar();
  };

  class B : class A{
      int b ;
      virtual void foo();
      virtual void quz();
  };
      • A的虚函数表中存放着foo和bar两个函数指针。
      • B的虚函数表中放继承自A的函数bar，也有重写的函数foo，还有新增的函数quz。
      • 所有虚函数地址都会存放在所属类的虚函数表vtbl中，另外在基类中声明为虚函数的成员方法，达到子类是仍然是虚函数，即使子类中重新定义基类虚函数时未使用virtual修饰，该函数地址仍会放在子类的虚函数表vtbl中。

    • 虚函数表是怎么构造和继承的?

      • 基类虚函数表的构造:按照虚函数声明的顺序将这些函数地址放入虚函数表中。
      • 子虚函数的构造和继承:将基类的虚函数表赋值到子类的虚函数表中，如果子类重写了父类的虚函数，则虚函数表就需要改变，否则不需要改变，如果新增了虚函数，将新增的函数地址加到虚函数表的后面。

    • 虚函数的调用过程，虚函数表示如何访问的?

        Base::virtual void show();
  Extend::virtual void show();

  Extend ext;
  Base *pBase = &ext;
  pBase->show();

      • 当调用pBase->show();的执行步骤:

        1. 判断base类中show是否是虚函数。
        2. 如果不是虚函数，寻找base::show()。如果是虚函数执行下一步。
        3. 访问pbase所指对象的虚函数表指针得到pbase所指对象所在类的虚函数表。
        4. 查找base中show()在声明的时候的位序x，到步骤三得到的虚函数表中找到位序x，从而得到要执行show函数的地址。
        5. 根据函数地址和base中声明的show的函数类型访问所指行的函数。

      • 多态产生的原因?
        如果子类没有重写父类的虚函数，那么子类虚函数表中存放的是父类的虚函数，如果重写了虚函数那么存放的就是重写后的虚函数，所以事先了多态的功能。

  • 如果父类中仅有方法，子类中只有一个int变量，那么sizeof大小是多少?

    • 父类的大小是1，子类的大小是4。类的实例化，所谓类的实例化就是在内存中分配一块地址，每个实例在内存中都有独一无二的地址。同样空类也会被实例化，所以编译器会给空类隐含的添加一个字节，这样空类实例化之后就有了独一无二的地址了。所以空类的sizeof为1。

  • 指针一定是4个字节大小吗?

    • 不一定，根据 不同的操作系统，指针的大小可能会不一样，64位才做系统指针的大小是8字节，而32位操作系统指针的大小只有4个字节。、

  • #define和inline函数的区别是什么?

    • 内联函数在编译时展开，而宏是由预处理器对宏进行展开
    • 内联函数会检查参数类型，宏定义不检查函数参数 ，所以内联函数更安全。
    • 宏不是函数，而inline函数是函数
    • 宏在定义时要小心处理宏参数，（一般情况是把参数用括弧括起来）

  • const static 在哪里初始化?

    • const定义的常量在超出其作用域之后其空间会被释放，而static定义的静态常量在函数执行后不会释放其存储空间.
    • static静态成员变量不能在类的内部初始化，在类的内部只是声明，定义必须在类定义体的外部，通常在类的实现文件中初始化，比如：double Account::Rate = 2.25，static关键字只能用于类定义体内部的声明中，并且在外部定义时不能标识为static
    • const成员变量也不能在类定义处初始化，只能通过构造函数初始化列表进行，并且必须有构造函数,const成员变量只在某个对象生存期内是常量，而对于整个类而言却可变的。因为类可以创建多个对象，不同的对象其const数据成员的值可以不同，所以不能在类的声明中初始化const数据成员，因为类的对象没有被创建时，编译器不知道const数据成员的值是什么。
    • const成员函数的主要目的是防止成员函数修改对象内容，即const成员函数不能修改成员变量的值，但是可以访问成员变量
    • static成员函数的主要目的是作为类作用域的全局函数，不能访问类的非静态数据成员，因为类的静态成员函数没有this指针，这导致：1.不能直接存取类的非静态成员变量，调用非静态成员函数，2.不能被声明为虚函数(没有this指针，实现不了多态，定义为虚函数没有意义）.
    • 类的static const变量必须在类外部进行初始化。

  • 派生类的构造函数和析构函数执行顺序?

    • 构造函数是基类行进行构造，派生类再进行构造，析构函数是派生类行进行析构，然后基类在进行析构。同时无需手动析构基类在析构派生类的时候回析构基类的。
    • 编译器会在派生类的析构函数中添加基类析构函数的语句，所以我们可以不写，其实也是调用了基类的析构函数。

  • 什么时候派生类类的析构函数不会被调用?

    • 当基类的析构函数不是虚函数的时候，并且一个基类的指针指向一个派生类的对象的时候，基类的析构函数不会被调用，发生内存泄漏。

  • 如何生成静态库和动态库?

    • 使用gcc生成静态库的方法：
      • 生成.o文件，使用gcc -c test.c -o test.o
      • 使用ar将test.o打包成为libtest.a文件，使用ar rcs -o libtest.a test.o
      • 编译main.c,并使用libtest.a静态文件，使用命令gcc -o app_static main.c -L. libtest.a或者gcc -o appstatic main.c libtest.a
    • 使用gcc生成动态库的方法:
      • 生成test.o目标文件，使用如下命令。在此处需要添加-fPIC参数，该参数用于生成位置无关代码已工生成动态库使用，使用命令：gcc -c -o test.o -fPIC test.c。
      • 使用-shared参数生成动态库，使用如下命令：gcc -shared -o libmyshare.so test.o ,上面的两个命令可以连起来用gcc -shared -fPIC -o libmyushared.so test.c
      • 编译main.c,使用动态链接库，使用如下命令:gcc -o app_share main.c -L. libmyshare.so

  • 如何使用gdb调试?

    • 新建一个源文件test.cpp.
    • 生成有可调试信息的可执行文件，g++ -g -o test test.cpp。
    • 启动gdb调试，gdb test
      • 查看源文件list or l,重复命令按回车键。
      • 设置调试断电break 16,查看断点信息info break。
      • 运行run。
      • 单步调试，step进入函数内部
      • 主过程调试，next
      • 查看变量，print a。
      • 设置变量的值: set var a=100
      • bt查看函数栈针和层级关系(backtrace)。
      • frame切换函数的栈针。
      • finish结束当前函数返回函数的调用点。
      • 继续到下一个断点continue。
      • 退出q

  • coredunmp查看core文件分析？

    • core文件是程序奔溃的时候产生的文件，里面会有变量的值，方便调试。
    • 为什么使用coredump：在编程的过程中，可能会产生段错误，也就是访问了错误的内存段，这时候编译错误不会提醒到哪一行，是的调试变得较为困难，这时候断错误存储的core错误文件就可以派上用场了。
    • 配置：直接在shell中输入ulimit -c unlimited,或者在.bashrc文件中添加ulimit -c unlimited
    • 产生段错误，访问非法内存，编译的时候要加上-g选项。
    • 查看core dump: gdb [exec file] [core file],使用bt命令查看程序运行到了那个地方。
    • bt可以显示函数的调用栈，f num可以打印num栈的信息。

  • 如何调试运行中的程序?

    • 编译带-g的选项。
    • 运行程序。
    • ps找到进程号。
    • 启动gdb，使用attach选项进程，这时候gdb会停止在程序的某处。
    • 按照gdb调试的方法调试，调试完成仍然可以使用run命令重启程序。

  • 运行了几天的程序放生错误，怎么解决问题?

    • 创建一个调试的版本并使用它，然后确保程序崩溃的时候创建了core文件，然后用gdb查看core文件发现问题所在。

  • 如何编写makefile?

    • makefile的好处:

      • 项目代码编译管理
      • 节省编译时间(只会编译修改过的文件)
      • 一次编写一直可以用。

        #目标:依赖(条件)
  #   命令
  #阶段一
  all:add.c sub.c dive.c mul.c main.c
      gcc add.c sub.c dive.c mul.c main.c -o app

  #阶段二

  dive.o:dive.c

  mul.o:mul.c

  main.o:main.c
      gcc -c main.c

  .PHONY:clean

  hello:

  #wildcard是内置函数寻找符合条件的文件。patsubst对文件名进行替换。

  • 编译器的编译过程？链接的时候做了什么事？在中间层优化时怎么做?

    • 编译器的编译过程：预处理，编译和优化,汇编，链接。
      • 预处理:将程序中引用的头文件包含到源代码中，并对一些宏进行替换。
      • 编译和优化:将用户可识别的语言翻译成一组处理器可以识别的操作码，生成目标文件，通常翻译成汇编语言。
      • 汇编:将汇编输出的汇编代码翻译成符合一定条件的机器代码，在linux系统一般表现为.obj文件。
      • 链接:将所有的.o文件系统的obj文件和动态库文件链接起来，就可以形成可执行文件了。

  • STL各容器如何实现?

    • vector:底层数据结构为数组，支持快速随机访问。
    • list:双向列表，支持快速增删。
    • deque(双端队列):底层数据结构为一个中央控制器器和多个缓冲区。支持收尾快速删除也支持随机访问。
    • stack:底层一般用list或deque实现，封闭头部即可，不用vector的原因是容量大小有限制，扩容耗时。
    • queue:底层一般用list或deque实现，封闭头部即可。
    • priority_queue的底层数据结构是vector,堆heap为处理规则来管理底层容器实现。
    • set底层为红黑树，有序，不可重复。
    • multiset底层红黑树，有序，可重复。
    • map底层为红黑树，有序，不重复。

{
    if(isFull()) {
        return false;
    }

    queue_[rear_] = data;
    rear_ = (rear_ + 1) % capacity_;

    return true;
}

{
    T data;

    if(isEmpty()) {
        throw std::out_of_range("Queue::dequeue() - Empty queue");
    }

    data = queue_[front_];
    front_ = (front_ + 1) % capacity_;

    return data;
}

template <typename T>
T Queue<T>::front()
{
    if(isEmpty()) {
        throw std::out_of_range("Queue::front() - Empty queue");
    }

    return queue_[front_];
}

{
    return (front_ == rear_);
}

{
    return (((rear_+1) % capacity_) == front_);
}

    return ((rear_ -  front_ + capacity_) % capacity_);

    • 平衡二叉树的定义是左右两颗子树的高度相差不为为1.
    • 平衡二叉树有四种不平衡的情况：
      • 对节点的左儿子的左子树进行一次插入。简称左左。
      • 对节点的左儿子的右子树进行一次插入。简称左右。
      • 对节点的右儿子的左子树进行一次插入。简称右左。
      • 对节点的右儿子的右子树进行一次插入。简称右右。
    • 调整措施：对于左左和右右的情况采用单旋转就可以解决，左左进行一次右旋操作，右右进程一次左旋操作。而对另外两种情况则进行两次旋转，对于左右的情况，先将节点的左孩子作为选装的根节点进行一次左旋转换为左左的情况，然后再进行一次右选即可。对于右左的情况，先对节点的右孩子进行一次右旋操作转换为右旋，然后在进行左旋即可。

  • 平衡二叉树的删除操作

    • 首先在整个二叉树中搜索要删除的节点，如果没有搜索到直接返回不做处理，否则指向一下操作:
    • 要删除节点是当前根节点t。
      • 如果左子树高于右子树，将左子树的最大值作为根节点，删除左子树最大值节点。
      • 如果左子树低于右子树，将右子树的最小值作为根节点，删除右子树最大值节点。
    • 如果要删除节点元素小于当前根节点t值，在左子树中进行删除，递归调用在左子树中实施删除。判断当期节点是否满足平衡条件，如果满足只需要更新节点高度信息，如果不满足，需要进行平衡的调整。
    • 如果删除节点大于当期节点t，在右子树进行茶树操作。

    3. LRU算法：最近最久未使用的意思。可以使用hashmap和链表实现，当需要插入新的数据的时候，如果新数据在链表中存在，则把该节点移到链表头部，如果不存在则新建一个节点放到链表头部，如果缓存满了删除链表中的的最后一个，在访问数据的时候如果数据在链表中存在，就把数据项移到链表的头部，否则返回-1，这样链表中的最后一个元素就是最近未访问的元素了。

  • 堆和栈的区别

    1. 申请方式：栈是系统自动分配不的，而堆的内存需要程序员自己申请。
    2. 申请后系统的响应:栈是系统为程序提供内存。堆是首先系统知道操作系统有一个记录空闲内存地址的链表，当系统收到程序的申请的时候，遍历该链表将第一个大于申请空间的堆节点找出，然后将该节点从空闲节点链表中删除，然后将该节点的空间分配给程序。
    3. 申请空间大小的限制:栈的最大容量是确定的。堆是从高地址扩展的数据结构，是不连续的内存区域。堆的大小受限于计算机系统中有效的虚拟内存。由此可见堆获得的空间比较灵活，也比较大。
    4. 申请效率的比较: 栈是系统自动分配速度较快，而对的速度较慢。容易产生内存碎片。
    5. 栈和堆中存储的内容:在函数调用的时候，第一个进栈的是主函数后的下一条指令的地址。然后是函数的各个参数，然后是函数中局部变量。堆一般是在头部用一个字节存储存放堆的大小。堆的具体内容有程序员安排。
    6. 一般栈的存储效率较为低。

  • TCP 的三次握手过程

    • listen:侦听来自远方的tcp端口的连接请求(服务器)
    • syn-sent:在发送连接请求后等待匹配的连接请求(客户端)
    • syn-recv:在收到和发送一个链接请求后等待对方对链接的确认(服务器)
    • established:代表一个打开的连接(服务器、客户端)
    • fin-wait-1:等待远程tcp连接中断请求，或先前连接中断请求的确认(客户端)
    • FIN-WAIT-2：从远程TCP等待连接中断请求（客户端）
    • CLOSE-WAIT：等待从本地用户发来的连接中断请求（服务器）
    • CLOSING：等待远程TCP对连接中断的确认（客户端、服务器)
    • LAST-ACK：等待原来的发向远程TCP的连接中断请求的确认（服务器）
    • TIME-WAIT：等待足够的时间以确保远程TCP接收到连接中断请求的确认（客户端）
    • CLOSED：没有任何连接状态（客户端、服务器）

    1. TCP服务器进程先创建传输控制块TCB, 时刻准备接受客户端进程的连接请求, 此时服务器就进入了 LISTEN（监听）状态。

    2. TCP客户端进程也是先创建传输控制块TCB, 然后向服务器发出连接请求报文，此时报文首部中的同步标志位SYN=1, 同时选择一个初始序列号 seq = x(x为随机生成), 此时，TCP客户端进程进入了 SYN-SENT（同步已发送状态）状态。TCP规定, SYN报文段（SYN=1的报文段）不能携带数据，但需要消耗掉一个序号。

    3. TCP服务器收到请求报文后, 如果同意连接, 则发出确认报文。确认报文中的 ACK=1, SYN=1, 确认序号是 x+1（x为第2步客户端发送过来的序列号）, 同时也要为自己初始化一个序列号 seq = y(y为随机生成), 此时, TCP服务器进程进入了SYN-RCVD（同步收到）状态。这个报文也不能携带数据, 但是同样要消耗一个序号。

    4. TCP客户端进程收到服务端回复后需要进行相关确认（该处需要检查第3步服务器发送的标志位ACK的值是否为1，确认序号是否为第2步客户端发送的初始序号x+1），当确认正确后，还要向服务器给出确认。确认报文的ACK=1，确认序号是 y+1，自己的序列号是 x+1。

    5. 此时，TCP连接建立，客户端进入ESTABLISHED（已建立连接）状态。当服务器收到客户端的确认后也进入ESTABLISHED状态，此后双方就可以开始通信了。

    • SYN表示建立连接，ACK表示响应，FIN表示关闭连接，PSH表示数据传输，RST表示链接重置。大写的值为0或1，而小写的ack和seq是小写的单词。

  • TCP 的拥塞控制是怎样的

    • 发送方维持一个拥塞窗口的状态变量，拥塞窗口的大小取决于网络的拥塞程度，并且在动态的改变，发送方让自己的发送窗口等于拥塞窗口，另外如果考虑接受放的接受能力，发送窗口可能小于拥塞窗口。慢开始算法的思路就是不要一开始就发送大量的数据，先测一下网络的拥塞程度，然后由小到大的增加网络的拥塞窗口。

    • 避免缓慢增长算法让拥塞窗口缓慢的增长，也就是每经过一个往返时间rtt就把发送方的拥塞窗口增加1。

    • 快重传要求接受方子啊手打搜一个失序的报文段后就立即发出重复确认，发送方只要一接收到三个重复确认就立刻重传未收到的报文段，而不必等待设置的计时器时间到。

    • 快重传配合的还有快恢复算法，当发送方连续收到三个重复确认是，就执行乘法减小算法，将ssthresh门限变为阻塞窗口的一般大小，但是接下去不执行慢开始算法。考虑到如果网络出现阻塞就不会收到几个重复确认的，所以发送方认为网络没有阻塞，所以不执行慢开始算法，而是将阻塞窗口设置为ssthresh的大小，然后执行拥色避免算法。

  • 304 状态码的意义？在 HTTP 协议中的实现

    • 当用户第一次请求index.html时，服务器会添加一个名为Last-Modified响应头，这个头说明了index.html的最后修改时间，浏览器会把index.html内容，以及最后响应时间缓存下来。当用户第二次请求index.html时，在请求中包含一个名为If-Modified-Since请求头，它的值就是第一次请求时服务器通过Last-Modified响应头发送给浏览器的值，即index.html最后的修改时间，If-Modified-Since请求头就是在告诉服务器，我这里浏览器缓存的index.html最后修改时间是这个，您看看现在的index.html最后修改时间是不是这个，如果还是，那么您就不用再响应这个index.html内容了，我会把缓存的内容直接显示出来。而服务器端会获取If-Modified-Since值，与index.html的当前最后修改时间比对，如果相同，服务器会发响应码304，表示index.html与浏览器上次缓存的相同，无需再次发送(节省传输成本)，浏览器可以显示自己的缓存页面，如果比对不同，那么说明index.html已经做了修改，服务器会响应200。

  • A 类地址和 B 类地址的区别

    • 一个A类IP地址是指， 在IP地址的四段号码中，第一段号码为网络号码，剩下的三段号码为本地计算机的号码。如果用二进制表示IP地址的话，A类IP地址就由1字节的网络地址和3字节主机地址组成，网络地址的最高位必须是“0”。A类IP地址中网络的标识长度为8位，主机标识的长度为24位。

    • 一个B类IP地址是指，在IP地址的四段号码中，前两段号码为网络号码。如果用二进制表示IP地址的话，B类IP地址就由2字节的网络地址和2字节主机地址组成，网络地址的最高位必须是“10”。B类IP地址中网络的标识长度为16位，主机标识的长度为16位。

    • 一个C类IP地址是指，在IP地址的四段号码中，前三段号码为网络号码，剩下的一段号码为本地计算机的号码。如果用二进制表示IP地址的话，C类IP地址就由3字节的网络地址和1字节主机地址组成，网络地址的最高位必须是“110”。C类IP地址中网络的标识长度为24位，主机标识的长度为8位。

    • A类地址适合大型网络，b类地址适合中型网络，c类地址适合小型网络。

  • memcached 的内存设计

    • Memcached 的内存分配以page为单位，默认情况下一个page是1M，可以通过-I参数在启动时指定。如果需要申请内存时，memcached会划分出一个新的 page并分配给需要的slab区域。page一旦被分配在重启前不会被回收或者重新分配
    • Memcached并不是将所有大小的数据都放在一起的，而是预先将数据空间划分为一系列slabs，每个slab只负责一定范围内的数据存储。如下图，每个slab只存储 大于其上一个slab的size并小于或者等于自己最大size的数据。例如：slab 3只存储大小介于137 到 224 bytes的数据。如果一个数据大小为230byte将被分配到slab 4中。从下图可以看出，每个slab负责的空间其实是不等的，memcached默认情况下下一个slab的最大值为前一个的1.25倍，这个可以通过修 改-f参数来修改增长比例。
    • Chunk 是一系列固定的内存空间，这个大小就是管理它的slab的最大存放大小。例如：slab 1的所有chunk都是104byte，而slab 4的所有chunk都是280byte。chunk是memcached实际存放缓存数据的地方，因为chunk的大小固定为slab能够存放的最大值， 所以所有分配给当前slab的数据都可以被chunk存下。如果时间的数据大小小于chunk的大小，空余的空间将会被闲置，这个是为了防止内存碎片而设计的。例如下图，chunk size是224byte，而存储的数据只有200byte，剩下的24byte将被闲置。
    • Memcached在启动时通过-m指定最大使用内存，但是这个不会一启动就占用，是随着需要逐步分配给各slab的。如果一个新的缓存数据要被存放，memcached首先选择一个合适的slab，然后查看该slab是否还有空闲的chunk，如果有则直接存放进去；如果没有则要进行申请。slab申请内存时以page为单位，所以在放入第一个数据，无论大小为多少，都会有1M大小的page被分配给该slab。申请到 page后，slab会将这个page的内存按chunk的大小进行切分，这样就变成了一个chunk的数组，在从chunk数组中选择一个用于存储 数据。如下图，slab 1和slab 2都分配了一个page，并按各自的大小切分成chunk数组。

    • tinyhttpd 是一个不到 500 行的超轻量型 Http Server，用来学习非常不错，可以帮助我们真正理解服务器程序的本质。

      • apache 发展到现在，模块超多，基本想到的都可以找到
      • apache 更为成熟，少 bug ，nginx 的 bug 相对较多

    • 操作的便利性：memcache数据结构单一，redis丰富一些，数据操作方面，redis更好一些，较少的网络IO次数，mongodb支持丰富的数据表达，索引，最类似关系型数据库，支持的查询语言非常丰富
    • 内存空间大小和数据量的大小: redis在2.0版本后增加了自己的VM特性，突破物理内存的限制；可以对key value设置过期时间（类似memcache）。memcache可以修改最大可用内存,采用LRU算法。mongoDB适合大数据量的存储，依赖操作系统VM做内存管理，吃内存也比较厉害，服务不要和别的服务在一起
    • 可用性： redis，依赖客户端来实现分布式读写；主从复制时，每次从节点重新连接主节点都要依赖整个快照,无增量复制，因性能和效率问题，所以单点问题比较复杂；不支持自动sharding,需要依赖程序设定一致hash 机制。一种替代方案是，不用redis本身的复制机制，采用自己做主动复制（多份存储），或者改成增量复制的方式（需要自己实现），一致性问题和性能的权衡。Memcache本身没有数据冗余机制，也没必要；对于故障预防，采用依赖成熟的hash或者环状的算法，解决单点故障引起的抖动问题。mongoDB支持master-slave,replicaset（内部采用paxos选举算法，自动故障恢复）,auto sharding机制，对客户端屏蔽了故障转移和切分机制。
    • 可靠性：redis支持（快照、AOF）：依赖快照进行持久化，aof增强了可靠性的同时，对性能有所影响。memcache不支持，通常用在做缓存,提升性能；MongoDB从1.8版本开始采用binlog方式支持持久化的可靠性。
    • 应用场景：redis：数据量较小的更性能操作和运算上。memcache：用于在动态系统中减少数据库负载，提升性能;做缓存，提高性能（适合读多写少，对于数据量比较大，可以采用sharding）。MongoDB:主要解决海量数据的访问效率问题。
